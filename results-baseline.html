🔑 Key Findings

N-max = 2 is optimal for all models

Across 9B, 27B, and 35B parameter models, the sweet spot for MTP draft tokens is n-max=2. Going beyond that either plateaus (27B) or starts hurting performance (35B n-max≥3).

Largest speedup at smallest scale

The 9B model sees a +39% generation speedup (48.0 vs 34.6 t/s). The 27B gets +45% (16.7 vs 11.5 t/s). The 35B only gains +11% (62.6 vs 56.5 t/s) — the benefit diminishes as model size grows.

N-max ≥ 5 is dangerous

The 9B model at n-max=8 drops to 22.5 t/s — a 35% degradation. Too many speculative tokens overwhelms the bandwidth. This applies across model sizes.

Vulkan auto-offloading beats manual --gpu-layers

Forcing all layers to GPU with --gpu-layers 99 hurts performance on larger models (27B dropped from 11.5 to 4.7 t/s). Let Vulkan auto-manage memory placement for best results.

Additional notes for Vulkan:

  • Do NOT set --gpu-layers N on Vulkan — let the backend auto-manage memory placement.
  • Avoid n-max ≥ 5 — especially dangerous on smaller models (9B at n=8 drops to -35%).
  • n-max = 2 is the sweet spot across all tested sizes — provides most of the benefit with minimal overhead.
  • Baseline generation counts are higher on larger models because prompt phase dominates (the model's KV cache fills faster at more tokens, and the larger parallel compute offsets per-token gen cost).
